# 本地自动化调度与自愈控制台 PRD

## 1. 文档目的

本文档用于指导前端原型设计与后续功能实现，重点说明产品目标、目标用户、核心业务流程、页面列表、页面模块、交互规则、空状态、加载状态、异常状态与后端接口依赖。

本文档基于当前项目代码反推，核心后端能力来自 FastAPI 网关、业务白名单、任务队列、任务查询、重试、自愈、截图证据与任务日志。

## 2. 产品概述

### 2.1 产品名称

暂定名：本地自动化调度与自愈控制台

### 2.2 产品定位

该产品是一个运行在本地环境中的自动化任务控制台，用于统一管理网页自动化任务和桌面自动化任务。

用户通过前端控制台完成以下工作：

- 查看本地自动化网关是否可用。
- 查看当前已注册、可调度的业务白名单。
- 提交单个自动化任务。
- 批量提交自动化任务。
- 查看任务执行状态、执行结果、失败原因和截图证据。
- 对已结束任务发起重试。
- 观察任务是否进入自愈流程以及自愈后是否成功重跑。

### 2.3 产品边界

本产品不是通用脚本执行器，也不是允许用户任意上传代码或传入 EXE 路径的自动化平台。

产品只调度后端白名单中已经注册的业务：

- 网页任务：通过已注册 Python 模块执行 Playwright 自动化。
- 桌面任务：通过已注册 AHK 编译 EXE 执行桌面自动化。

前端不提供直接编辑业务源码、上传脚本、修改白名单、修改 `.env` 或管理本地浏览器 profile 文件的能力。

## 3. 目标用户

### 3.1 运营执行人员

运营执行人员需要稳定触发固定自动化流程，例如搜索、采集、表单处理、桌面工具调用等。

核心诉求：

- 不关心脚本细节，只需要选择业务并提交任务。
- 能快速知道任务是否成功。
- 失败时能把任务 ID、截图、错误信息交给维护人员。

### 3.2 自动化开发者

自动化开发者负责编写、注册和维护业务脚本。

核心诉求：

- 查看业务是否已经接入网关。
- 验证任务参数是否正确。
- 查看失败堆栈、截图、源码路径和原始请求。
- 判断重试和自愈是否生效。

### 3.3 本地自动化维护者

维护者负责保证本地网关、浏览器上下文、日志目录和自愈流程正常运行。

核心诉求：

- 确认网关健康状态。
- 快速定位失败任务。
- 查看任务生命周期。
- 判断失败是业务参数问题、脚本问题、环境问题还是外部页面变化。

## 4. 核心概念

### 4.1 业务

业务是后端白名单中显式注册的自动化能力。

业务字段：

- `name`：业务名称。
- `kind`：业务类型，取值为 `web` 或 `desktop`。
- `module`：网页任务对应的 Python 模块。
- `executable`：桌面任务对应的 EXE 路径。
- `source`：可审计、可自愈关注的业务源码路径。

### 4.2 任务

任务是一次具体的自动化执行请求。

任务字段：

- `task_id`：任务唯一 ID。
- `status`：任务状态。
- `request`：原始请求参数。
- `attempts`：已经执行的尝试次数。
- `healed`：是否完成过自愈修复。
- `parent_task_id`：如果该任务来自手动重试，则记录原任务 ID。
- `created_at`：创建时间。
- `started_at`：开始时间。
- `finished_at`：结束时间。
- `result`：业务标准返回结果。
- `error_traceback`：失败堆栈。
- `business_source`：业务源码路径。
- `screenshot`：失败截图路径。

### 4.3 任务状态

任务状态包括：

- `queued`：已提交，等待执行。
- `running`：正在执行。
- `healing`：执行失败后正在尝试自愈。
- `succeeded`：执行成功。
- `failed`：执行失败。

### 4.4 自愈

自愈是任务在普通重试失败后触发的自动修复流程。

当前后端逻辑：

- 任务必须开启 `enable_self_healing`。
- 任务必须有对应业务源码路径。
- 普通执行和重试均失败后进入 `healing`。
- 自愈流程会收集源码路径、错误堆栈、截图、原始请求和任务结果。
- 如果修复器返回成功，网关会重新加载业务并自动重跑一次。
- 自愈不会无限循环。

## 5. 后端接口依赖

### 5.1 健康检查

接口：`GET /health`

用途：

- 判断网关是否在线。
- 首屏展示服务状态。

成功响应结构：

```json
{
  "code": 0,
  "msg": "success",
  "data": {
    "status": "ok",
    "service": "automation-gateway"
  },
  "screenshot": null
}
```

### 5.2 获取业务列表

接口：`GET /api/v1/businesses`

用途：

- 获取可提交任务的业务白名单。
- 作为提交任务页的业务选择数据源。

响应字段：

- `name`
- `kind`
- `module`
- `executable`
- `source`

### 5.3 提交单个任务

接口：`POST /api/v1/tasks`

用途：

- 创建任务并进入后台队列。
- 接口返回 `202`，不等待任务执行完成。

请求结构：

```json
{
  "kind": "web",
  "business": "demo_search",
  "params": {
    "query": "关键词",
    "limit": 5,
    "profile": "demo"
  },
  "args": [],
  "timeout_seconds": 300,
  "max_retries": 1,
  "enable_self_healing": true
}
```

字段规则：

- `kind`：必填，`web` 或 `desktop`。
- `business`：必填，必须是已注册业务。
- `params`：网页任务参数。
- `args`：桌面任务参数数组。
- `timeout_seconds`：默认 300，范围 1 到 3600。
- `max_retries`：默认 1，范围 0 到 5。
- `enable_self_healing`：默认 true。

### 5.4 批量提交任务

接口：`POST /api/v1/tasks/batch`

用途：

- 一次提交多个任务。
- 每个任务独立排队、执行、重试、查询。
- 批量数量范围为 1 到 100。

请求结构：

```json
{
  "tasks": [
    {
      "kind": "web",
      "business": "demo_search",
      "params": {
        "query": "关键词",
        "limit": 5,
        "profile": "demo"
      },
      "args": [],
      "timeout_seconds": 300,
      "max_retries": 1,
      "enable_self_healing": true
    }
  ]
}
```

### 5.5 查询任务详情

接口：`GET /api/v1/tasks/{task_id}`

用途：

- 查询任务状态、尝试次数、结果、失败证据。
- 任务详情页轮询该接口直到任务进入终态。

### 5.6 重试任务

接口：`POST /api/v1/tasks/{task_id}/retry`

用途：

- 基于原任务请求创建一个新任务。
- 原任务保持不可变。

规则：

- 只有已结束任务可以重试。
- `queued`、`running`、`healing` 状态不可重试。
- 重试成功后返回新任务记录，新任务的 `parent_task_id` 指向原任务。

## 6. 当前后端缺口

以下能力不是当前代码已有接口，但若要实现完整控制台体验，建议补充：

### 6.1 任务列表接口

建议接口：`GET /api/v1/tasks`

用途：

- 总览页展示最近任务。
- 支持任务列表筛选、搜索和状态统计。

建议查询参数：

- `status`
- `business`
- `kind`
- `keyword`
- `limit`
- `offset`

### 6.2 任务事件接口

建议接口：`GET /api/v1/tasks/{task_id}/events`

用途：

- 展示任务生命周期时间线。
- 读取任务对应 JSONL 事件日志。

### 6.3 截图访问接口

建议接口：`GET /api/v1/files/screenshots/{filename}`

用途：

- 前端可直接预览截图。
- 避免只展示本地绝对路径。

### 6.4 业务参数 Schema 接口

建议在业务列表中增加 `schema` 字段，或新增接口：`GET /api/v1/businesses/{name}/schema`

用途：

- 前端根据业务自动生成参数表单。
- 避免所有业务都只能手写 JSON。

## 7. 信息架构

建议导航结构：

```text
控制台
├── 总览
├── 业务
├── 新建任务
├── 批量提交
├── 任务查询
├── 运行环境
└── API 调试
```

MVP 阶段可以先实现：

```text
控制台
├── 总览
├── 业务
├── 新建任务
└── 任务详情
```

其中任务详情可以通过任务 ID 直接进入，不依赖完整任务列表接口。

## 8. 页面需求

## 8.1 总览页

### 页面目标

让用户进入系统后快速判断本地网关是否可用、当前有哪些任务状态、是否有失败任务需要处理。

### 页面模块

#### 8.1.1 服务状态区

展示内容：

- 网关状态：在线 / 离线。
- 服务名。
- 最近检测时间。
- 检测失败原因。

交互：

- 点击刷新按钮重新请求 `/health`。
- 健康检查失败时展示启动提示。

#### 8.1.2 状态统计区

展示内容：

- 排队中任务数。
- 运行中任务数。
- 自愈中任务数。
- 成功任务数。
- 失败任务数。

说明：

- 当前后端没有任务列表接口时，该区域只能统计当前前端会话内提交过的任务。
- 接入任务列表接口后，应统计后端持久化任务。

#### 8.1.3 最近任务区

展示字段：

- 任务 ID。
- 业务名称。
- 类型。
- 状态。
- 尝试次数。
- 是否自愈。
- 创建时间。
- 结束时间。
- 操作。

操作：

- 查看详情。
- 复制任务 ID。
- 对终态任务发起重试。

#### 8.1.4 快捷入口区

展示内容：

- 新建网页任务。
- 新建桌面任务。
- 批量提交。
- 输入任务 ID 查询。

### 空状态

- 没有任务时展示：暂无任务记录，可以从新建任务开始。
- 网关离线时展示：无法连接本地自动化网关，请确认服务已启动。

### 加载状态

- 首次进入页面时，服务状态卡片显示加载中。
- 最近任务区显示表格骨架。

### 异常状态

- `/health` 请求失败时，状态显示离线。
- 如果任务统计接口缺失，显示当前会话统计并提示数据范围。

## 8.2 业务列表页

### 页面目标

展示当前系统允许调度的业务白名单，帮助用户确认可以提交哪些任务。

### 页面模块

#### 8.2.1 筛选区

筛选条件：

- 全部。
- 网页任务。
- 桌面任务。

搜索条件：

- 业务名。
- 模块路径。
- 源码路径。

#### 8.2.2 业务表格

字段：

- 业务名。
- 类型。
- Python 模块。
- EXE 路径。
- 源码路径。
- 操作。

操作：

- 提交任务。
- 复制业务名。
- 复制源码路径。
- 查看详情。

#### 8.2.3 业务详情抽屉

展示内容：

- 业务名。
- 业务类型。
- 模块路径。
- EXE 路径。
- 源码路径。
- 建议请求结构。

对于示例业务 `demo_search`，展示示例参数：

```json
{
  "query": "关键词",
  "limit": 5,
  "profile": "demo"
}
```

### 空状态

当业务列表为空时：

- 展示文案：暂无已注册业务。
- 辅助说明：新业务需要先在后端业务白名单中注册。

### 加载状态

- 表格显示骨架行。
- 筛选器禁用。

### 异常状态

- 业务列表加载失败时，展示错误原因和重试按钮。

## 8.3 新建任务页

### 页面目标

让用户基于已注册业务创建单个自动化任务。

### 页面模块

#### 8.3.1 业务选择区

字段：

- 任务类型：`web` / `desktop`。
- 业务名称：从业务列表中选择。

交互：

- 切换任务类型后，业务名称下拉只展示对应类型业务。
- 选择业务后，右侧或下方展示业务摘要。

#### 8.3.2 参数配置区

网页任务：

- 展示 `params` JSON 编辑器。
- 支持格式化 JSON。
- 支持恢复示例 JSON。

桌面任务：

- 展示 `args` 参数数组编辑器。
- 支持新增、删除、调整参数顺序。

#### 8.3.3 执行设置区

字段：

- 超时时间，默认 300 秒。
- 最大重试次数，默认 1。
- 开启自愈，默认开启。

规则：

- 超时时间范围：1 到 3600。
- 最大重试次数范围：0 到 5。

#### 8.3.4 请求预览区

实时展示最终提交给后端的 JSON。

用途：

- 方便开发者检查请求结构。
- 支持复制请求 JSON。

#### 8.3.5 提交结果区

提交成功后展示：

- 任务 ID。
- 初始状态。
- 创建时间。
- 查看详情按钮。
- 复制任务 ID 按钮。

### 关键交互

- JSON 格式错误时禁用提交按钮。
- 未选择业务时禁用提交按钮。
- 提交中按钮显示 loading。
- 提交成功后不清空表单，便于继续提交类似任务。
- 提交成功后自动把该任务加入当前会话最近任务。

### 空状态

- 没有可选业务时，提示暂无对应类型业务。

### 加载状态

- 加载业务列表时禁用业务选择器。
- 提交中禁用提交按钮和关键输入。

### 异常状态

- 422 参数校验失败：展示字段级错误或完整错误 JSON。
- 404 未注册业务：提示该业务不存在或类型不匹配。
- 400 配置错误：展示后端返回信息。
- 500 服务器错误：展示错误摘要和重试提交按钮。

## 8.4 批量提交页

### 页面目标

让用户一次提交多个自动化任务，并逐条查看提交结果。

### 页面模块

#### 8.4.1 批量编辑区

能力：

- 新增任务。
- 复制任务。
- 删除任务。
- 折叠任务。
- 展开任务。

每个任务包含：

- `kind`
- `business`
- `params`
- `args`
- `timeout_seconds`
- `max_retries`
- `enable_self_healing`

#### 8.4.2 JSON 导入区

能力：

- 粘贴完整批量请求 JSON。
- 格式化 JSON。
- 校验 JSON。
- 从 JSON 生成任务条目。

#### 8.4.3 批量校验区

展示内容：

- 总任务数。
- 校验通过数量。
- 校验失败数量。
- 失败任务位置和原因。

#### 8.4.4 提交结果区

每条结果展示：

- 序号。
- 是否提交成功。
- 任务 ID。
- 错误信息。
- 查看详情。

### 关键交互

- 最少 1 条任务，最多 100 条任务。
- 任一任务本地校验失败时，默认禁止提交。
- 提交后逐条展示后端结果。
- 某些任务提交失败时，不影响其他成功任务。

### 空状态

- 没有任务时展示：添加任务或粘贴批量 JSON。

### 加载状态

- 提交中显示批量提交进度。

### 异常状态

- 整体请求失败时展示全局错误。
- 单条任务失败时展示在对应行内。

## 8.6 任务详情页

### 页面目标

展示单个任务的完整状态、请求、结果、失败证据和操作。

### 页面模块

#### 8.6.1 顶部摘要区

展示内容：

- 任务 ID。
- 状态。
- 业务名称。
- 业务类型。
- 尝试次数。
- 是否自愈。
- 父任务 ID。

操作：

- 刷新。
- 复制任务 ID。
- 重试任务。

#### 8.6.2 状态时间区

展示内容：

- 创建时间。
- 开始时间。
- 结束时间。
- 执行耗时。

规则：

- 如果任务未开始，开始时间显示为空。
- 如果任务未结束，结束时间显示为空。
- 执行中任务展示持续时间。

#### 8.6.3 原始请求区

展示内容：

- `kind`
- `business`
- `params`
- `args`
- `timeout_seconds`
- `max_retries`
- `enable_self_healing`

能力：

- JSON 展示。
- 复制 JSON。
- 折叠/展开。

#### 8.6.4 执行结果区

展示内容：

- `code`
- `msg`
- `data`
- `screenshot`

成功任务：

- 优先展示业务数据摘要。
- 同时保留完整 JSON。

失败任务：

- 优先展示错误消息。
- 引导用户查看失败证据。

#### 8.6.5 失败证据区

仅当任务失败或有错误信息时重点展示。

展示内容：

- 错误堆栈 `error_traceback`。
- 失败截图 `screenshot`。
- 业务源码路径 `business_source`。
- 原始失败结果 `result`。

能力：

- 复制错误堆栈。
- 复制源码路径。
- 查看截图。
- 复制完整失败 JSON。

#### 8.6.6 自愈信息区

展示内容：

- 是否开启自愈。
- 是否进入自愈状态。
- 是否 `healed=true`。
- 自愈后重跑结果。

状态说明：

- `healing`：当前正在自愈。
- `healed=true` 且 `succeeded`：自愈后重跑成功。
- `healed=true` 且 `failed`：自愈执行过，但重跑仍失败。
- `healed=false` 且 `failed`：未完成自愈或自愈未修复。

### 关键交互

- 当状态为 `queued`、`running`、`healing` 时，页面自动轮询。
- 当状态进入 `succeeded` 或 `failed` 后，停止自动轮询。
- 用户可以手动刷新。
- 重试按钮只在终态任务上可用。
- 点击重试后弹窗确认，说明重试会创建新任务，不会修改原任务。
- 重试成功后提示新任务 ID，并提供跳转到新任务详情。

### 空状态

- 无截图：本任务没有生成截图。
- 无错误堆栈：本任务没有错误堆栈。
- 无结果：任务尚未产生执行结果。

### 加载状态

- 首次进入详情页显示骨架。
- 自动轮询时显示最后刷新时间。
- 重试中禁用重试按钮。

### 异常状态

- 404：任务不存在。
- 409：任务仍在执行中，不能重试。
- 500：网关异常。

## 9. 任务提交表单规则

### 9.1 通用规则

- 必须选择任务类型。
- 必须选择业务。
- 业务必须来自白名单接口。
- 超时时间必须大于 0 且不超过 3600。
- 最大重试次数必须在 0 到 5 之间。
- 是否自愈为布尔值。

### 9.2 网页任务规则

- 使用 `params` 作为主要参数。
- `args` 默认为空数组。
- `params` 必须是合法 JSON 对象。

### 9.3 桌面任务规则

- 使用 `args` 作为主要参数。
- `params` 默认为空对象。
- `args` 必须是字符串数组。

## 10. 示例业务需求

当前项目内置示例业务：`demo_search`。

业务类型：`web`

功能：打开 Bing，执行搜索，并返回搜索结果标题。

参数：

- `query`：必填，搜索关键词。
- `limit`：可选，默认 5，范围建议 1 到 20。
- `profile`：可选，默认 `demo`，用于区分浏览器持久登录态。

成功返回数据：

```json
{
  "query": "关键词",
  "count": 5,
  "titles": ["结果标题 1", "结果标题 2"],
  "url": "https://www.bing.com/search..."
}
```

前端在选择 `demo_search` 时，可以提供上述示例参数。

## 11. 全局交互规则

### 11.1 状态展示

所有任务状态必须使用一致的标签：

- queued：排队中。
- running：运行中。
- healing：自愈中。
- succeeded：成功。
- failed：失败。

### 11.2 刷新规则

- 总览页手动刷新为主。
- 任务详情页在非终态时自动轮询。
- 自动轮询间隔建议 2 到 5 秒。
- 终态后停止轮询。

### 11.3 复制规则

以下内容必须支持复制：

- 任务 ID。
- 业务名。
- 请求 JSON。
- 响应 JSON。
- 错误堆栈。
- 源码路径。
- 截图路径。

### 11.4 重试规则

- 重试不会修改原任务。
- 重试会创建新任务。
- 重试成功后应引导用户查看新任务。
- 非终态任务不可重试。

### 11.5 时间展示规则

- 所有后端时间为 ISO 字符串。
- 前端展示应转换为本地时间。
- 鼠标悬停或详情中保留原始 ISO 时间。

## 12. 空状态规范

### 12.1 无业务

标题：暂无已注册业务

说明：当前网关没有返回可调度业务，请先在后端白名单中注册业务。

操作：刷新业务列表。

### 12.2 无任务

标题：暂无任务记录

说明：提交任务后可以在这里查看状态和结果。

操作：新建任务。

### 12.3 无查询结果

标题：未找到任务

说明：请检查任务 ID 是否正确。

操作：重新输入。

### 12.4 无截图

标题：没有截图

说明：该任务没有生成截图，或截图路径不可访问。

### 12.5 无错误

标题：没有错误信息

说明：该任务当前没有错误堆栈。

## 13. 加载状态规范

### 13.1 页面首次加载

- 展示页面级骨架。
- 不使用整页空白。
- 服务状态和业务列表可以独立加载。

### 13.2 表格加载

- 展示 3 到 5 行骨架。
- 保留表头。

### 13.3 表单提交

- 提交按钮显示 loading。
- 禁止重复提交。
- 表单内容保持可见。

### 13.4 任务轮询

- 展示最后刷新时间。
- 不遮挡详情内容。
- 刷新失败时保留上一次成功数据。

## 14. 异常状态规范

### 14.1 网关离线

触发条件：

- `/health` 请求失败。
- 其他接口网络错误。

展示内容：

- 网关离线。
- 无法连接本地自动化服务。
- 提示检查服务是否启动。

### 14.2 请求参数错误

触发条件：

- HTTP 422。

展示内容：

- 请求参数错误。
- 展示字段级错误详情。

### 14.3 业务不存在

触发条件：

- HTTP 404。

展示内容：

- 业务或任务不存在。
- 建议刷新业务列表或检查任务 ID。

### 14.4 任务不可重试

触发条件：

- HTTP 409。

展示内容：

- 当前任务仍在执行中，不能手动重试。

### 14.5 后端异常

触发条件：

- HTTP 500。

展示内容：

- 后端执行异常。
- 展示后端返回 `msg`。
- 保留完整响应 JSON 供复制。

## 15. 权限与安全约束

当前产品默认运行在本地可信环境，但仍应遵守以下约束：

- 前端不得允许用户输入任意 Python 模块路径。
- 前端不得允许用户输入任意 EXE 路径。
- 前端不得提供直接修改业务白名单的能力。
- 前端不得直接编辑 `.env`。
- 前端不得上传或执行任意脚本。
- 所有任务必须基于后端返回的业务白名单提交。
- 截图和源码路径展示时应以只读形式呈现。

## 16. MVP 范围

### 16.1 MVP 必做

- 健康检查。
- 业务列表。
- 单任务提交。
- 任务 ID 查询。
- 任务详情。
- 任务状态轮询。
- 失败堆栈展示。
- 截图路径展示。
- 手动重试。
- 请求和响应 JSON 复制。

### 16.2 MVP 可延后

- 批量提交可视化编辑器。
- API 调试页。
- 运行环境页。
- 完整任务列表。
- 任务事件时间线。
- 截图在线预览。
- 业务参数 Schema 自动表单。

### 16.3 后端未补齐前的前端降级方案

如果没有任务列表接口：

- 总览页只展示网关状态、业务数量和当前会话任务。
- 用户通过新建任务页提交后，前端在本地缓存任务 ID。
- 用户可以通过任务查询页输入任务 ID 查看详情。

如果没有截图访问接口：

- 任务详情只展示截图绝对路径。
- 提供复制路径能力。

如果没有业务参数 Schema：

- 参数区先使用 JSON 编辑器。
- 对内置示例业务提供手写示例参数。

## 17. 原型设计重点

给 Stitch 或其他原型设计工具时，建议优先表达以下模块：

- 左侧导航和顶部服务状态。
- 业务列表表格。
- 新建任务表单。
- JSON 参数编辑器。
- 请求预览。
- 提交成功结果。
- 任务详情顶部状态摘要。
- 任务结果 JSON 面板。
- 失败证据面板。
- 重试确认弹窗。

原型中不需要重点表达：

- 强品牌视觉。
- 营销型首页。
- 复杂权限系统。
- 脚本编辑器。

## 18. 验收标准

### 18.1 业务列表

- 用户可以看到所有后端注册业务。
- 用户可以按业务类型筛选。
- 用户可以从业务直接进入新建任务。

### 18.2 新建任务

- 用户可以选择业务并提交任务。
- JSON 参数错误时不能提交。
- 提交成功后能看到任务 ID。
- 用户可以跳转到任务详情。

### 18.3 任务详情

- 用户可以看到任务状态、请求、结果和时间信息。
- 非终态任务会自动刷新。
- 失败任务能看到错误堆栈和截图路径。
- 终态任务可以重试。

### 18.4 异常处理

- 网关离线时有明确提示。
- 任务不存在时有明确提示。
- 参数错误时有明确提示。
- 重试冲突时有明确提示。

### 18.5 审计与复制

- 用户可以复制任务 ID。
- 用户可以复制请求 JSON。
- 用户可以复制结果 JSON。
- 用户可以复制错误堆栈。
- 用户可以复制源码路径和截图路径。

## 19. 非目标

当前阶段不做：

- 用户登录和权限系统。
- 云端部署管理。
- 任意脚本上传。
- 在线编辑业务源码。
- 在线修改业务白名单。
- 在线管理浏览器 profile。
- 在线查看完整本地日志文件。
- 复杂图表分析。

## 20. 结论

本产品的前端核心不是装饰性门户，而是一个面向本地自动化任务的可观测控制台。

设计和实现应优先保证：

- 任务能提交。
- 状态能看清。
- 失败能定位。
- 证据能复制。
- 终态能重试。
- 自愈过程能被用户理解。

在后端接口尚未补齐任务列表、事件日志和截图访问能力之前，前端应优先完成单任务闭环，并通过任务 ID 查询与当前会话任务缓存实现可用的 MVP。
